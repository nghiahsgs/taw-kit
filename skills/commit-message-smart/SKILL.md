# commit-message-smart — Read the Diff, Write the Truth

## Step 0 — Detect repo convention

```bash
# inspect last 20 commits to learn project style
git log -20 --pretty=format:'%s' 2>/dev/null
```

Classify the observed style:

| Pattern seen in history | Output style |
|---|---|
| `feat(scope): ...` / `fix(scope): ...` | Conventional Commits (full, with scope) |
| `feat: ...` / `fix: ...` | Conventional Commits (no scope) |
| `Add X`, `Fix Y`, `Update Z` | Imperative title-case (old GitHub style) |
| mix / chaos | Default to Conventional Commits — nudge team to standard |

Use the detected style for output. Don't force Conventional if project uses something else.

## Step 1 — Gather diff context

```bash
git diff --staged 2>/dev/null || git diff 2>/dev/null
git diff --staged --stat 2>/dev/null
git diff --staged --name-only
```

If nothing staged, warn: "Chưa stage file nào. Stage trước (vd `git add <file>`) rồi gõ lại." Stop.

## Step 2 — Classify the change (type)

Scan the diff. Pick ONE type based on **what the diff actually does**, not just file names:

| Type | Triggers |
|---|---|
| `feat` | NEW functionality — new functions, new exports, new routes, new UI elements |
| `fix` | Bug fix — diff adds guards, null-checks, corrects wrong logic/values, changes test assertions to pass |
| `refactor` | Rename, extract, move, restructure — same behaviour, different shape. NO behaviour change |
| `perf` | Bundle size reduction, lazy loads, memoization, caching, query optimization |
| `test` | Only `*.test.*` / `*.spec.*` files touched |
| `docs` | Only `*.md`, comments, JSDoc changes |
| `style` | Whitespace, formatting, semicolons — Prettier-style with zero logic change |
| `build` | package.json deps, lockfile, Dockerfile, tsconfig — infra of the build |
| `ci` | `.github/workflows/`, `.circleci/`, `ci` scripts |
| `chore` | Everything else housekeeping — config tweaks, ignore files, rename of files |
| `revert` | Explicit `Revert "..."` — usually auto-generated by git revert |

**Edge cases:**
- Mixed types → prefer the dominant one by line count, but if two categories are both substantial → suggest `split into multiple commits` rather than picking one. Don't hide a `feat` inside a `fix`.
- Generated types / lockfile changes alone → `chore`. If alongside a `feat`, stays `feat`.
- Test file changes alongside code change → usually `feat` or `fix` (tests support the main change).

## Step 3 — Pick the scope

From file paths, extract the deepest meaningful folder:

| Files touched | Scope |
|---|---|
| `app/checkout/page.tsx`, `app/checkout/success/page.tsx` | `checkout` |
| `components/ui/button.tsx` | `ui` |
| `lib/stripe.ts`, `app/api/webhooks/stripe/route.ts` | `stripe` |
| `supabase/migrations/*.sql` | `db` |
| `app/(marketing)/page.tsx` | `marketing` |
| Tests across codebase | omit scope or `test` |
| Scattered, no common folder | omit scope |

If diff touches more than 3 separate scopes, omit scope — it's a cross-cutting change.

Scope should be lowercase, kebab-case, 1-2 words max.

## Step 4 — Write the subject

Rules:
1. **Imperative mood** — "add login" not "added login" / "adds login"
2. **No period at end**
3. **Max 72 chars** (soft limit 50 for most readable)
4. **Describe the WHY or the OUTCOME, not the WHAT** — the diff already shows the what
5. **Lowercase first letter** (after `type(scope): `)

| Bad | Good | Why |
|---|---|---|
| `fix(auth): fixed bug` | `fix(auth): reject expired magic links before session create` | Specific — reader knows what class of bug |
| `feat(ui): updated button` | `feat(ui): add loading+disabled state to primary button` | Concrete outcome |
| `refactor: cleanup` | `refactor(cart): extract calcTotal into pure function` | Names the extraction |
| `chore: stuff` | `chore(deps): bump next 14.2 → 15.0 with async cookies codemod` | Reason for bump |

## Step 5 — Body (only if needed)

Add a body if:
- Breaking change (prefix line with `BREAKING CHANGE:` per Conventional Commits)
- Non-obvious reason — issue reference, incident pointer
- Multiple related changes need listing

Skip body for straightforward commits (>80% of cases). Less is more.

Template:
```
feat(checkout): add Stripe Checkout session for one-time buy

Closes #234. The Polar SDK we tried first doesn't support VN bank transfer,
switched to Stripe for broader VN payment method coverage. Polar remains
available behind a feature flag.

BREAKING CHANGE: POLAR_ACCESS_TOKEN env var replaced by STRIPE_SECRET_KEY.
Update .env.local before deploy.
```

## Step 6 — Render options to user

Show 1 recommended + 1-2 alternatives (VN if user VN):
```
Đề xuất commit message:

✓ feat(checkout): add Stripe Checkout for one-time purchase

Khác (nếu muốn):
  feat(stripe): wire checkout session with success URL
  feat(payment): switch primary provider to Stripe

Gõ:
  1   → dùng message số 1 (recommended)
  2|3 → chọn số khác
  edit <text> → viết tay
  split → chia commit nhỏ hơn (diff có nhiều type)
```

## Step 7 — Hand off

After user picks, pass message to `git-auto-commit` skill:
```
Skill({ skill: "git-auto-commit", args: "<chosen message>" })
```

Or run directly if user wants to stay in this skill:
```bash
git commit -m "<message>"
```

## Gotchas

- **Giant diffs (>500 lines)** → sampling strategy: read `--stat`, first 100 lines of each touched file's diff. Ask user to summarize intent if still unclear.
- **Claude's urge to write essay subjects** → hard cap 72 chars. Chop and move context to body.
- **"WIP" / "stuff" temptation** — refuse. Either commit is ready with real message, or save and stage smaller chunks.
- **Auto-generated files** (build output, types regen) mixed with manual changes → suggest user unstage generated files or split commits.
- **Forgetting language** — if user writes VN elsewhere, keep prompts VN but commit SUBJECT always in English (convention — easier for tools / GitHub).

## Constraints

- Commit subject ALWAYS in English (convention) — even if user interacts in VN
- Body can be VN if project/team uses VN in commits (check via history)
- Never commit secrets inadvertently — warn if diff includes `.env*`, `*.key`, `*.pem`
- Never force user to Conventional if history uses something else
- Breaking changes ALWAYS need `!` or `BREAKING CHANGE:` footer — don't let user skip
- Max 72 chars on subject. Hard. Truncate or reject.
